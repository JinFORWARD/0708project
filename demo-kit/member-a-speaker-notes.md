## 3. 03 第 1 节：实验范围怎么讲

03 第 1 节要突出两个关键词：**本地授权** 和 **网关入口**。

我的讲法：

这里先说明安全边界。我们的攻击工具只允许访问 `127.0.0.1`、`localhost` 或 `::1`，也就是本机自己访问自己。这样不会影响公网或第三方系统。

本实验里，Nginx 扮演网关入口，Python 后端只负责返回简单响应。我们要观察的是：当入口层遇到慢速请求头时，默认配置和加固配置有什么差异。

背后的项目依据：

- `attack/slowloris.py` 里限制了 `ALLOWED_HOSTS = {"127.0.0.1", "localhost", "::1"}`，所以攻击工具默认只打本地。
- `backend/app.py` 是一个很小的本地 HTTP 服务，不涉及真实业务系统。

## 4. 03 第 2 节：为什么选 Slowloris

这一节我不要讲太复杂，只讲它和“网关入口控制”的关系。

我的讲法：

Slowloris 的特点不是大流量，而是“慢”。它先发一部分 HTTP 请求头，但一直不发结束空行。Nginx 会认为这个客户端还没发完，于是保留连接继续等待。

所以这个攻击更像是在考验入口层的连接生命周期管理，而不是考验后端业务代码有没有漏洞。也就是说，攻击可能还没到后端，网关入口就已经被拖住了。

我自己的理解：

- 普通请求：请求头完整结束，Nginx 可以转发给后端。
- Slowloris：请求头一直不结束，Nginx 先卡在读取请求头阶段。
- 防御方向：不要无限等，要设置超时和连接限制。

## 5. 03 第 3 节：实验架构怎么讲

这一节看图讲，不用展开源码。

链路是：

```text
Slowloris 工具 / 正常请求 -> Nginx 网关 127.0.0.1:8080 -> Python 后端 127.0.0.1:18080
```

我的讲法：

这里最重要的是攻击脚本和正常请求都先访问 Nginx 的 8080 端口，不直接访问后端。后端在 18080 端口，只在 Nginx 后面提供 `/health` 和简单响应。

这符合真实系统里的网关位置：外部请求一般先经过入口层，比如 Nginx、API（Application Programming Interface，应用程序编程接口）网关或负载均衡，再到业务服务。

背后的项目依据：

- `backend/app.py` 里 `/health` 返回 HTTP 200，用来模拟正常用户健康检查。
- `nginx-baseline.conf` 和 `nginx-hardened.conf` 都把请求转发到 `127.0.0.1:18080`。
- `attack/slowloris.py` 默认目标是 `127.0.0.1:8080`，也就是只打网关入口。

## 7. 03 第 5 节：攻击现象记录怎么讲

这是我最应该讲清楚的实测部分。讲的时候只抓两个对照：**基线组慢连接持续存活；加固组慢连接被快速清理。**

### 7.1 基线组讲法

03 里的关键数据：

```text
60 个慢连接
10 秒补发一次头部
持续 35 秒
4 轮都是 alive=60
TCP established 约 62
/health 7 次都是 200
access.log 最后出现 400，request_time 约 40 秒
```

我的讲法：

基线组使用的是 `nginx-baseline.conf`，它只做反向代理，没有专门针对 Slowloris 做防护。实验里攻击工具打开 60 个慢连接，测试持续 35 秒，4 轮检查都是 `alive=60`。

这说明慢连接一直活着，Nginx 没有很快把未完成请求头清理掉。同时 TCP（Transmission Control Protocol，传输控制协议）已建立连接稳定在 62 左右，说明入口连接资源确实被占住。

要特别注意：`/health` 7 次都是 200，不代表没有攻击现象。因为本次是低强度本机实验，目标不是把服务打挂，而是观察慢连接生命周期。核心现象是：慢连接在基线配置下能持续占用入口。

### 7.2 加固组讲法

03 里的关键数据：

```text
第 1 轮 alive=60
第 2 轮 alive=0
failed_since_last_round=60
TCP established 从 62 降到 1-2
access.log 出现 408
request_time 约 5 秒
```

我的讲法：

加固组使用 `nginx-hardened.conf`。同样是 60 个慢连接，第一轮还能看到 `alive=60`，第二轮就变成 `alive=0`，说明服务端已经把慢连接关掉了。

同时，TCP 已建立连接从 62 快速下降到 1-2，Nginx access log（访问日志）出现 408，`request_time` 约 5 秒。这和加固配置里的 `client_header_timeout 5s` 对上了。

所以加固组最明显的效果是：Nginx 不再长期等待未完成请求头，而是在约 5 秒后超时清理。

### 7.3 第 5 节一句话总结

基线组证明“慢连接能占住入口”；加固组证明“请求头超时能把慢连接快速释放”。

## 8. 03 第 6 节：Slowloris 原理怎么讲

这一节可以用新人话，不要讲成攻击教程。

我的讲法：

正常 HTTP 请求头最后会有一个空行，也就是 `\r\n\r\n`，表示请求头结束。Slowloris 的做法是先发请求行和部分请求头，然后每隔一段时间补发一小段头部，但一直不发结束空行。

这样 Nginx 会觉得“客户端还在发请求头”，所以连接不能立即释放。如果这样的连接很多，入口层就会被拖住。

可以用一个比喻：

这像有人去窗口办业务，每次只说半句话，不把材料交完整。窗口不能结束这单业务，只能继续等。这样的人多了，后面正常办业务的人就可能排队。

背后的项目依据：

- `attack/slowloris.py` 的 `send_initial_header` 发送了请求行和几个请求头，但没有发送空行结束。
- `send_keepalive_header` 会继续补发 `X-Lab-...` 这类小头部，让连接保持“还没发完”的状态。

## 9. 03 第 7 节：防御配置怎么讲

这一节是成员 A 的重点。不要背全部配置，抓住“基线没有专门防护，加固增加超时和连接限制”。

### 9.1 基线配置

我的讲法：

基线配置只做本地反向代理：客户端请求进入 Nginx，再转发到 Python 后端。它不是错误配置，只是没有专门针对慢速请求头做限制，所以用作对照组。

### 9.2 加固配置

03 中核心配置：

```nginx
client_header_timeout  5s;
client_body_timeout    10s;
keepalive_timeout      10s;
reset_timedout_connection on;
limit_conn_zone $binary_remote_addr zone=perip:10m;
limit_conn_status 429;
limit_conn perip 10;
```

我的讲法可以这样拆：

| 配置 | 我怎么解释 |
| --- | --- |
| `client_header_timeout 5s` | 客户端 5 秒内不把请求头发完整，Nginx 就不继续等。本次最关键。 |
| `client_body_timeout 10s` | 请求体发送太慢也会被限制，更多是为 Slow POST（慢速 HTTP POST 请求体攻击）扩展。 |
| `keepalive_timeout 10s` | 空闲连接不要保留太久，减少无效占用。 |
| `reset_timedout_connection on` | 超时连接尽快断开。 |
| `limit_conn_zone` / `limit_conn perip 10` | 按 IP（Internet Protocol，互联网协议）限制并发连接，避免单个来源占太多入口连接。 |
| `limit_conn_status 429` | 如果连接超限，用 429 记录，便于日志识别。 |

本次最明显生效的是 `client_header_timeout 5s`，因为 Slowloris 正好攻击的是“请求头迟迟不结束”。

如果被问 `limit_conn` 为什么不是主导现象：

> 因为 5 秒请求头超时更早触发，慢连接先被超时清理掉了，所以日志里主要看到 408，而不是 429。

## 10. 03 第 8 节：防御前后对比怎么讲

这一节适合做阶段总结。

我的讲法：

对比看，基线组 60 个慢连接在 35 秒测试期间持续存活，已建立连接稳定在 62 左右；加固组约 5 秒触发请求头超时，第二轮检查时慢连接已经全部关闭，已建立连接降到 1-2。

两组 `/health` 都是 200，所以这次实验不是为了把服务打挂。真正的差异是慢连接生命周期：基线让慢连接持续占用入口，加固把未完成请求头限制在约 5 秒内，从而释放连接。

一句话总结：

**防御配置的价值不是让攻击脚本不存在，而是缩短慢连接占用网关资源的时间。**

## 13. 如果只给我 3 分钟

大家好，我负责环境搭建、防御配置和架构说明。

这次实验是本地授权环境，链路是 Slowloris 工具或正常请求先访问 `127.0.0.1:8080` 上的 Nginx 网关，再由 Nginx 转发到 `127.0.0.1:18080` 的 Python 后端。

Slowloris 的核心不是大流量，而是请求头一直不结束。正常 HTTP 请求头最后应该有空行表示结束，但 Slowloris 会隔一段时间补一点头部字段，让 Nginx 以为请求还没发完，于是持续保留连接。

我们做了基线和加固两组对照。基线配置只做反向代理，不加专门防护。测试中 60 个慢连接在 35 秒内一直存活，攻击日志 4 轮都是 `alive=60`，TCP 已建立连接稳定在 62 左右。这说明入口连接被慢请求占住了。

加固配置里最关键的是 `client_header_timeout 5s`，意思是客户端 5 秒内不把请求头发完整，Nginx 就不继续等。加固后同样的 60 个慢连接，第二轮检查时变成 `alive=0`，TCP 已建立连接从 62 快速下降到 1-2，access.log 出现 408，`request_time` 约 5 秒。

所以本次结论是：正常 `/health` 请求在两组下都能返回 200，但真正的差异在慢连接生命周期。基线组让慢连接持续占用入口连接，加固组把未完成请求头限制在约 5 秒内，从而更快释放 Nginx 资源。

## 14. 最容易被问的问题

### 问题 1：你怎么知道攻击现象真的出现了？

答：基线组里，攻击脚本开了 60 个慢连接，35 秒测试期间 4 轮都是 `alive=60`，同时 TCP 已建立连接稳定在 62 左右。这说明未完成请求头连接一直占着 Nginx 入口。

### 问题 2：为什么 `/health` 都是 200，还能说有攻击效果？

答：这次是低强度本机实验，不是为了把服务打挂。`/health` 仍然 200 说明普通请求在本轮压力下还能成功；真正要看的攻击现象是慢连接生命周期。

### 问题 3：加固后为什么能说明防御成功？

答：同样的 60 个慢连接，在加固组里第二轮就全部失败，TCP 已建立连接从 62 快速降到 1-2。Nginx access.log 还出现 408，`request_time` 约 5 秒，对应 `client_header_timeout 5s`。

### 问题 4：`client_header_timeout` 是什么？

答：它控制 Nginx 等待客户端发送完整请求头的最长时间。Slowloris 的核心就是一直不发完整请求头，所以把这个等待时间设短，就能减少慢连接长期占用入口资源。

### 问题 5：400、408、429 分别是什么意思？

答：400 是请求不完整或格式异常，基线组里主要是攻击脚本最后主动关闭连接后出现；408 是请求超时，加固组里对应请求头读取超时；429 是请求或连接过多，本次配置里设置了连接超限时返回 429，但这轮主要现象不是它。

### 问题 6：这个实验和网关课程有什么关系？

答：它说明网关不只是转发请求，也负责入口层资源管理。攻击者不一定要打到后端，只要让入口层长期等待，就可能影响系统可用性。所以网关要做超时、连接限制和日志观测。

### 问题 7：为什么不现场跑？

答：这轮实验结果已经由 agent 在本机自动跑过，并写入日志和短报告。现场脚本复现暂时不够稳定，为了避免展示时间被环境问题打断，本次以固定实验参数、已有日志和报告结论为准。
