# 成员 A 讲解稿：环境搭建、防御配置、架构整理

定位：成员 A 负责把实验环境讲清楚，让大家知道流量从哪里来、经过哪里、Nginx（高性能 Web 服务器、反向代理和网关软件）做了什么防御。

建议时长：5 分钟左右。

## 1. 开场

大家好，我负责这次实验里的环境搭建、防御配置和架构整理。我们做的是一个本地授权实验，不访问外部系统。整体链路是：正常客户端或攻击脚本先访问 Nginx 网关，再由 Nginx 转发到 Python（解释型编程语言）后端服务。

可以用一句话解释本实验：我们想验证慢速请求头会不会占住网关连接，以及 Nginx 加上防护参数后能不能更快释放这些连接。

## 2. 架构说明

本地端口分成两层：

| 组件 | 地址 | 我的解释 |
| --- | --- | --- |
| Python 后端 | `127.0.0.1:18080` | 真正返回业务响应的服务。 |
| Nginx 网关 | `127.0.0.1:8080` | 对外入口，负责转发、记录日志和做连接控制。 |
| Slowloris 工具 | 连接 `127.0.0.1:8080` | 只打本机网关，不打外部地址。 |

我会强调：后端服务没有直接暴露给攻击脚本，攻击脚本看到的是 Nginx 这个入口。这符合网关在真实系统里的位置。

## 3. 基线配置

基线配置文件是：

```text
nginx/conf/nginx-baseline.conf
```

它只做反向代理，不加慢速攻击防护。这里可以解释 HTTP（HyperText Transfer Protocol，超文本传输协议）请求会先进入 Nginx，再被转发给后端。

我的讲法：基线配置不是坏配置，它只是没有专门针对 Slowloris 做限制，所以我们用它当对照组。

## 4. 防御配置

加固配置文件是：

```text
nginx/conf/nginx-hardened.conf
```

重点讲这些参数：

| 参数 | 新人版解释 |
| --- | --- |
| `client_header_timeout 5s` | 客户端 5 秒内不把请求头发完整，Nginx 就不继续等。 |
| `client_body_timeout 10s` | 请求体发送太慢也会被限制，主要面向 Slow POST（慢速 HTTP 请求体攻击）扩展。 |
| `keepalive_timeout 10s` | 空闲连接不要保留太久。 |
| `reset_timedout_connection on` | 超时连接尽快清理。 |
| `limit_conn perip 10` | 按 IP（Internet Protocol，互联网协议）限制单个来源可占用的连接数。 |

我会重点说：本次实测里最明显生效的是 `client_header_timeout 5s`，因为 Slowloris 的核心就是请求头迟迟不结束。

## 5. 实测结论

基线组里，60 个慢连接在 35 秒内持续存活，TCP（Transmission Control Protocol，传输控制协议）已建立连接稳定在 62 左右。

加固组里，第一轮还能看到 60 个连接，第二轮就变成 0，Nginx 日志里出现了约 5 秒的 408 超时记录。

我的总结句：防御配置的价值不是让攻击脚本消失，而是把慢连接占用网关资源的时间从几十秒缩短到约 5 秒。

## 7. 可能被问到的问题

问：为什么不用公网目标测试？
答：这是安全边界要求。攻击工具只允许本地地址，避免误伤第三方系统。

问：为什么没有用很大的连接数？
答：这是培训实验，目标是理解原理和验证防御，不是压垮本机。

问：`limit_conn` 为什么不是本次最明显的现象？
答：因为 `client_header_timeout 5s` 更早触发，把未完成请求头先清理掉了，所以这轮主要看到的是 408 超时。
