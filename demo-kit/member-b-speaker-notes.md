# 成员 B 讲解稿：攻击工具、指标采集、现象分析

定位：成员 B 负责把 Slowloris（慢速 HTTP 请求头攻击）的原理、攻击脚本、指标和现象讲清楚。

建议时长：5 分钟左右。

## 1. 开场

我负责攻击模拟和可观测性指标。这里的攻击只在本机实验环境里运行，目标是 `127.0.0.1:8080`，也就是成员 A 搭好的 Nginx 网关。

我们没有选择大流量压测，而是选择 Slowloris，因为它更能体现“慢速连接占住入口资源”的特点。

## 2. 慢速攻击原理

慢速攻击有几类：

| 类型 | 简单解释 | 本次是否实现 |
| --- | --- | --- |
| Slowloris | 慢慢发送 HTTP（HyperText Transfer Protocol，超文本传输协议）请求头，一直不发送结束空行。 | 已实现。 |
| Slow POST | 请求头正常发完，但请求体慢慢发。 | 未实现，作为扩展。 |
| TLS 慢握手 | TLS（Transport Layer Security，传输层安全协议）握手阶段慢慢拖。 | 未实现，作为扩展。 |

本次选 Slowloris，是因为它最直接对应 `client_header_timeout` 这个 Nginx 防御参数。

## 3. 攻击脚本怎么做

攻击脚本位置：

```text
attack/slowloris.py
```

它做三件事：

1. 连接本机 Nginx：`127.0.0.1:8080`。
2. 发送一部分 HTTP 请求头，但不发送空行结束符。
3. 每隔一段时间补发一小段头部，让 Nginx 以为请求还没发完。

我的讲法：这不是靠大流量打满带宽，而是让入口层一直等，从而占住连接资源。

## 4. 我们采集了哪些指标

完整指标清单在：

```text
observability/metrics-checklist.md
```

本次重点看三类：

| 指标 | 说明 |
| --- | --- |
| 攻击工具 alive 数 | 看慢连接是否还存活。 |
| TCP（Transmission Control Protocol，传输控制协议）连接数 | 看 8080 端口入口连接是否被占住。 |
| Nginx 日志状态码和关键词 | 看连接是被客户端关闭，还是被 Nginx 超时清理。 |

此外还有 `/health` 正常请求，用来确认普通访问是否还能返回 200。

## 5. 基线组现象

基线配置下，攻击脚本打开 60 个慢连接。4 轮检查都显示：

```text
alive=60
failed_since_last_round=0
```

同时 TCP 已建立连接稳定在 62 左右。这说明慢连接在测试期间一直占着入口。

日志里最后出现 400，约 40 秒，是因为测试结束时攻击脚本主动关闭连接。我的理解是：基线组没有很快把它们清理掉。

## 6. 加固组现象

加固配置下，同样参数再跑一次。第一轮还是 60 个连接，但第二轮变成：

```text
alive=0
failed_since_last_round=60
```

Nginx access log（访问日志）出现约 5 秒的 408，error log（错误日志）出现读取请求头超时。这说明 Nginx 主动判断“这个请求头太慢了”，然后释放连接。

## 8. 收尾总结

我的总结句：Slowloris 的危险点在于“慢”，不是“流量大”。所以防御也不是只看带宽，而是要看连接生命周期、请求头读取超时、异常状态码和入口连接数。

## 9. 可能被问到的问题

问：为什么 `/health` 没有明显变慢？
答：本次是低强度本机实验，目的是验证慢连接生命周期。连接数继续升高时，正常请求更可能排队或失败。

问：为什么看 TCP 连接数？
答：Slowloris 主要占用连接，所以 TCP 已建立连接数是最直接的观测指标之一。

问：后端为什么不一定收到请求？
答：因为请求头没有结束，Nginx 还没把完整请求转发给后端，压力主要卡在网关入口层。
